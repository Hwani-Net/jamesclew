# 증상
어댑터 stream 변환 vs 모델 변경 같은 **순수 엔지니어링 판단**을 "옵션 ①/②/③으로 어떻게 진행하시겠습니까?"식으로 사용자에게 결정 위임. 사용자: "이런 판단은 나보다 외부 모델 자문 구하는게 더 낫잖아. 판단의 기준을 잘 설정해."

# 원인
PITFALL-121(자체 escalation으로 사용자 결정 뒤집기) 직후 보수적으로 모든 결정을 사용자에 위임. 결정 경계를 잘못 설정.

# 결정 분류 (재정의)

## 사용자 결정 (필수 위임)
- **활성 프리셋 변경** — 비용/품질 트레이드오프 정책
- **유료 업그레이드** — 돈 지출
- **외부 서비스 가입/해지** — 계정 영향
- **방향성/우선순위** — "어느 프로젝트 먼저"
- **데이터 삭제** — 복구 불가 작업

## 외부 모델 자문 (엔지니어링)
- **버그 진단** — 어댑터 stream 변환 버그 등
- **아키텍처 선택** — 라이브러리/패턴 비교
- **코드 리뷰** — 보안/성능
- **디버깅 우선순위** — 어디부터 파볼지
- **모델/툴 선택** — 작업에 최적인 도구

## 자체 판단 (즉시 실행)
- **검증/테스트** — curl 호출, 로그 확인
- **재시도/롤백** — 실패한 작업 복구
- **모니터링** — 헬스체크, 산출물 확인
- **사실 수집** — 파일 읽기, grep

# 해결
1. PITFALL-121 + 122 동시 적용:
   - 사용자 결정 = 비용/방향성/돈
   - 외부 모델 = 엔지니어링 판단
   - 자체 = 검증/실행
2. 엔지니어링 결정 분기점 도달 시 즉시 Codex/GPT-4.1 자문 → 결과를 사용자에게 보고
