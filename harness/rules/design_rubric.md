# Design Rubric — Evaluator 참조 기준

출처: Anthropic Harness 실험 (Opus 4.6 기준, 2026-04)
용도: `/qa` Evaluator가 프런트엔드 결과물을 등급 평가할 때 "정답" 기준으로 삼음

## 평가 원칙
- Generator ≠ Evaluator (자기 검수 편향 방지)
- 각 축마다 0-10 점수. 8점 이상이면 PASS, 6-7 REWORK, 5 이하 FAIL
- 총점보다 **최저축**을 우선 (한 축이라도 FAIL이면 재작업)

## 4대 평가축

### 1. Design 일관성 (Consistency) — Claude 약점
**질문**: 화면이 통일된 시스템을 따르는가, 아니면 요소만 따로 붙여 놓은 것인가?

| 점수 | 기준 |
|-----|------|
| 10 | 전 페이지에서 spacing/color/radius/shadow 단 하나의 토큰 체계 |
| 8 | 컴포넌트 간 간격·정렬 일관. 색상 팔레트 유지 |
| 6 | 버튼 스타일은 같지만 카드/입력 필드 간격 제각각 |
| 4 | 섹션마다 다른 디자인 언어. 카피-페이스트 흔적 |
| 0 | 체계 없음. AI가 매 요소를 새로 그림 |

**감지 신호**:
- CSS variable 없이 하드코딩된 색상/간격 → -2
- Tailwind 유틸리티만 사용하며 디자인 토큰 없음 → -1

---

### 2. 독창성 (Originality) — Claude 가장 큰 약점
**질문**: 사람이 의도를 가지고 디자인한 것처럼 보이는가?

| 점수 | 기준 |
|-----|------|
| 10 | 명확한 비주얼 아이덴티티. 카피/컬러/레이아웃이 제품 본질을 반영 |
| 8 | 템플릿을 벗어난 선택 1-2개 (독특한 레이아웃 or 컬러) |
| 6 | 무난한 모던 디자인. 다른 SaaS와 구별 안 됨 |
| 4 | **보라색+흰색 그라데이션**, 블러 원, 센터 정렬 카드 (AI 기본형) |
| 0 | ChatGPT/Claude가 처음 뽑은 그대로 |

**AI 클리셰 블랙리스트** (발견 즉시 -3):
- `bg-gradient-to-r from-purple-500 to-pink-500` 계열
- 대형 blur circle 배경 장식
- "Get Started" + 이메일 입력 히어로 섹션 템플릿
- 3-column feature grid with generic icons
- Lucide/Heroicons만 사용 (커스텀 일러스트 0)

**가점 신호**:
- 브랜드 고유 컬러 (채도/명도 선택에 의도) → +2
- 비대칭 레이아웃 또는 비표준 그리드 → +2
- 커스텀 일러스트/사진 → +3

---

### 3. 완성도 (Polish) — Claude 약점
**질문**: 디테일이 기술적으로만 맞는가, 창의적으로 맞춰졌는가?

| 점수 | 기준 |
|-----|------|
| 10 | Typography 계층·행간·자간, 대비율, 간격 리듬 모두 완벽 |
| 8 | 전 항목 기본기 통과 + 1-2개 세심한 터치 |
| 6 | WCAG AA 대비율은 맞지만 Typography 계층이 평평 |
| 4 | 글자 크기 2개만 사용. 간격 리듬 없음 |
| 0 | 기본 브라우저 스타일에 가까움 |

**체크리스트**:
- [ ] Typography 계층 최소 4단 (h1/h2/h3/body)
- [ ] 행간 1.4-1.6 (본문), 1.1-1.2 (제목)
- [ ] 색상 대비율 WCAG AA (4.5:1)
- [ ] 간격이 8px 또는 4px 그리드 준수
- [ ] Hover/Focus/Active 상태 구분

---

### 4. 기능성 (Functionality) — Claude 이미 잘함
**질문**: 각 컴포넌트가 사용자 경험에 시각적으로 기여하는가?

| 점수 | 기준 |
|-----|------|
| 10 | 모든 인터랙션이 목적이 명확. 불필요한 장식 0 |
| 8 | 주요 플로우는 명확. 일부 부가 요소는 장식적 |
| 6 | 기능은 작동하지만 시각적 피드백 부족 |
| 4 | 버튼 상태/로딩/에러 처리 누락 |
| 0 | 작동하지 않음 |

**Playwright 검증 항목**:
- [ ] 주요 CTA 클릭 → 기대 동작
- [ ] 폼 입력 → 유효성 피드백
- [ ] 로딩 상태 시각화
- [ ] 에러 상태 복구 경로

## Evaluator 사용법

```bash
# /qa 커맨드 내부에서 호출
cat D:/jamesclew/harness/rules/design_rubric.md | \
  codex exec "다음 rubric으로 {URL}을 평가하라. 각 축별 점수(0-10)와 근거, 최저축, PASS/REWORK/FAIL 판정을 JSON으로 출력"
```

## 통과 기준
- **PASS**: 4개 축 모두 8점 이상
- **REWORK**: 1-2개 축이 6-7점 (수정 후 재평가)
- **FAIL**: 어느 축이라도 5점 이하 (Generator로 반송)

## 코드 프로젝트 확장
프런트엔드가 아닌 백엔드/라이브러리 프로젝트는 아래 축 사용:
1. **아키텍처 일관성** — 레이어 분리, 의존성 방향
2. **API 설계 독창성** — REST/GraphQL 관용구 벗어난 의도
3. **테스트 완성도** — 커버리지 + 엣지 케이스
4. **기능성** — E2E 통과
