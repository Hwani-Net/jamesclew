# /prd — PRD 생성 (Product Requirements Document)

프로젝트의 PRD를 체계적으로 생성합니다.
대표님의 아이디어 한 줄 → 완벽한 기획서로 변환.

> ⚠️ **Planner 고수준화 원칙 (Anthropic 2026-04 연구 기반)**
> Opus 4.6부터는 **마이크로 태스크 분할이 오히려 해가 됨** — 작은 오류 하나가 9단계 뒤로 전파되어 에이전트가 스스로 방향 수정 불가.
> PRD는 **제품 레벨**만 다룬다: 무엇을 만들 것인가 (What), 왜 만드는가 (Why), 누구를 위해 (Who).
> **기술 구현 디테일은 금지** — How는 에이전트가 스스로 찾음.

## 사전 조건
- 프로젝트 폴더에서 실행 (package.json 또는 기획 의도 필요)
- `/pipeline-install` 전에 실행하면 CLAUDE.md에 PRD가 통합됨

## 실행 절차

### 1. 입력 수집 (대화형 — 반드시 질문할 것)
사용자의 프롬프트에서 추출 시도:
- 프로젝트명
- 한 줄 설명 (무엇을 만드는가)
- 타겟 사용자 (누구를 위한 것인가)
- 핵심 동기 (왜 만드는가)

**사용자가 한 줄만 입력해도 정상.** 아래 절차를 따름:
1. 프롬프트에서 추출 가능한 정보를 정리하여 보여줌
2. 빠진 항목 + 구체화가 필요한 항목을 **1회 질문** (최대 3개 질문 묶어서)
3. 사용자 답변 수신 후 리서치 단계로 진행
4. 사용자가 "알아서 해" 또는 무응답이면 리서치 결과로 자체 판단하여 진행

**질문 없이 바로 PRD 작성 금지.** 최소 1회 사용자 확인을 거쳐야 함.
단, 사용자가 프롬프트에서 이미 기능·대상·동기를 모두 명시했으면 확인만 하고 바로 진행.

### 2. 자동 리서치 (Tavily/Perplexity)
PRD 작성 전 반드시 시장 조사를 실행:
- **경쟁사 벤치마킹**: 유사 서비스 3-5개 조사 (기능, UI, 가격, 평점)
- **사용자 페인포인트**: 경쟁사 리뷰/커뮤니티에서 불만 수집
- **차별화 기회**: 경쟁사가 못하는 것, 사용자가 원하는 것

### 3. PRD 생성
아래 템플릿에 맞춰 `PRD.md` 파일을 프로젝트 루트에 생성.

---

## 4. PRD 자기수정 루프 (최대 3라운드)

생성된 PRD를 아래 관점으로 자체 검증:

| 관점 | 체크 항목 |
|------|----------|
| 사용자 | 페인포인트가 구체적인가? 추측이 아닌 조사 기반인가? |
| 제품 | Job Story와 기능이 1:1 매핑되는가? |
| 고수준 | **기술 구현 디테일이 없는가?** (파일/함수/라이브러리 이름 금지) |
| 제품 | Acceptance Criteria가 "사용자가 X할 수 있음"인가? |
| 디자인 | 벤치마킹 레퍼런스가 있는가? 디자인 토큰이 명시되었는가? |
| 스코프 | 기능 5개, 태스크 15개 상한선을 지키는가? |

- blocker 0개 → APPROVED
- blocker 있음 → 수정 → 다음 라운드
- 최대 3라운드 후 강제 APPROVED (무한 루프 방지)

## 5. 결과 저장
- `PRD.md` → 프로젝트 루트
- CLAUDE.md에 PRD 요약 섹션 추가 (있으면)
- 옵시디언에 사본 저장: `$OBSIDIAN_VAULT/02-projects/[프로젝트명]/PRD.md`
